# Clarify Brief: session-continuity

**Slug:** session-continuity
**Timestamp:** 20260414T160000Z
**Status:** draft
**Complexity:** standard

---

## Idea

session continuity — two problems: 1. capture working context before compaction or /clear fires so the session can be resumed without loss, 2. write a durable session summary at exit that Claude reads at next session start via MEMORY.md injection. parent context: dev-memory-feel dossier establishes that session-start is the highest-ROI injection moment and CLAUDE.md-adjacent MEMORY.md is the right surface.

---

## Goal

A developer who exits a Claude Code session — whether through /clear, context compaction, or normal exit — finds a MEMORY.md in the project root at their next session start that tells them exactly where they were and what mattered, without having to ask for it.

---

## Non-Goals

- Not a redesign of the extraction pipeline or window classifier
- Not about improving F1 scores or extraction recall
- Not about cross-project memory or global context (project-scoped only)
- Not about general/personal memory (lifestyle, preferences) — only working context and session state
- Not a UI for browsing or searching the full vault — only the session-start injection surface
- Not about vault storage format changes — reads from existing facts.db

---

## Constraints

- Local embeddings must remain local (privacy invariant — no embeddings sent to API)
- Vault operates independently from Cortex state
- Must work within Claude Code's session model — no persistent process between sessions
- Must not modify developer-authored CLAUDE.md files — system writes to a separate MEMORY.md
- Any injection must be readable and editable by the developer (no invisible writes)
- Must work with the existing window classifier output — no new extraction infrastructure required for MVP
- Haiku for any LLM summarization calls (cost constraint for high-frequency events)

---

## Assumptions

- Claude Code exposes a hook that fires before or at compaction (PreCompact or equivalent) — needs verification
- Claude Code exposes a Stop/session-exit hook that fires on normal exit and /clear
- The window classifier already extracts enough episodic and learning facts to construct a useful session summary without a dedicated summarization pass
- MEMORY.md at the project root will be read by Claude at session start because it is in the same directory as CLAUDE.md — needs verification against Claude Code's context loading behavior
- The developer will not delete MEMORY.md manually between sessions; if they do, the next session simply has no injection (graceful degradation)
- Compaction and /clear are different events with different intent: compaction = context management mid-session (keep working), /clear = deliberate reset (may or may not want continuity)

2. (q2/codebase) Read current session-start behavior — check CLAUDE.md files, any existing hook scripts, and confirm whether MEMORY.md in project root would be auto-read.
3. (q3/codebase) Map window classifier output categories (episodic 120, learning 341, procedural 70) against session summary content requirements — what's already captured vs what needs a dedicated extraction pass.
