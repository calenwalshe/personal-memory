# Clarify Brief: memory-phenomenology

**Slug:** memory-phenomenology
**Timestamp:** 20260414T120000Z
**Status:** draft
**Complexity:** complex

---

## Idea

i think we need to talk more about the type of memory we want this to have rather than the plumbing itself... what the memory enables rather than how it's extracted, what the memory will lead to in its effect... the phenomenology of the design...

---

## Goal

Define what the personal memory system should *feel like* to use — articulating the experiential outcomes and behavioral effects it should produce — so that all technical decisions (extraction strategy, storage grain, recall UX, forgetting policy) can be evaluated against this standard rather than against implementation metrics alone.

---

## Non-Goals

- Not redesigning the extraction pipeline or changing window_classifier.py
- Not proposing specific implementation changes to any existing script
- Not a product spec or user story backlog
- Not evaluating whether the current F1 = 0.25 is good or bad (that is downstream)
- Not defining what memory *is* technically — only what it *does experientially*

---

## Constraints

- Local embeddings must remain local (privacy invariant from system-map: embeddings never sent to API)
- Vault operates independently from Cortex `.cortex/state.json` — memory is not a Cortex phase artifact
- Must work within Claude Code's session model — no persistent process between sessions; memory is loaded, not running
- Any memory the system holds must be inspectable and deletable by the user — no opaque recall

---

## Assumptions

- The current extraction-first design (optimizing F1) may not be optimizing for the right goal; a high-F1 extractor can still produce a bad phenomenological experience
- "Good memory" means different things at different time horizons — what matters in a 10-minute session differs from what matters across 6 months
- The primary felt value of memory is not *information retrieval* but *being known* — the experience of a collaborator who understands your context without being re-briefed
- There is a failure mode worse than forgetting: confidently wrong memory that misleads rather than helps
- Privacy is first-class — the user must trust that memory of sensitive details is controlled, not passively accumulated

---

## Open Questions

- q1 (mechanism): What is the primary phenomenological failure mode — forgetting something important, misremembering, or remembering the wrong things?
- q2 (factual): What should the user feel at session start when memory is working well — what is the ideal opening moment?
- q3 (factual): At what grain should memory operate: atomic facts, narrative threads, or contextual patterns?
- q4 (comparison): How do existing memory-enabled AI systems (Letta, Mem0, Zep, Rewind) describe the user experience they're targeting?
- q5 (landscape): What human memory research (episodic vs. semantic memory, forgetting curves, the feeling of familiarity) is applicable to this design?
- q6 (codebase): Does the current F1 metric on window classification map to any phenomenological goal — what would a phenomenologically grounded success metric look like?
- q7 (factual): What is the right forgetting curve — should stale or low-signal memories decay, and what does that feel like to the user when it happens?
- q8 (factual): Should memory be visible and inspectable, or work invisibly in the background — and what are the trust implications of each?

---

## Next Research Steps

1. (q4/comparison) Survey how Letta, Mem0, Zep, Rewind, and similar systems describe the user experience and phenomenological goals of their memory layer — not the architecture, the intended felt effect.
2. (q5/landscape) Survey human memory phenomenology literature: episodic vs. semantic memory, the feeling of familiarity vs. recollection, forgetting curves, source monitoring — what maps to AI assistant memory design?
3. (q1/mechanism) Identify the documented failure modes users most frequently report with AI memory systems — and which failure mode damages trust most irreversibly.
4. (q3 + q7/factual) Investigate what memory grain and decay policies other systems have settled on, and how users describe the experience when those policies are right vs. wrong.
5. (q6/codebase) Audit the current system: run `/recall` on several real topics, observe what it surfaces and how it reads — map the current felt experience against the goal articulated in this brief.
