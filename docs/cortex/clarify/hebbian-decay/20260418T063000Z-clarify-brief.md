# Clarify Brief: hebbian-decay

**Slug:** hebbian-decay
**Timestamp:** 20260418T063000Z
**Status:** draft
**Complexity:** standard

---

## Idea

Decay (most important — blocking validity). Right now weights only go up. Without a decay term, the graph becomes a noise accumulator — every entity pair you ever co-mentioned eventually reaches MAX_WEIGHT regardless of whether the association is still active. Biological Hebbian has "use it or lose it." Experiment: weight *= (1 - DECAY) per session where a pair didn't fire. Validates that the graph is encoding current interest topology, not just historical co-occurrence.

---

## Goal

Add a per-session decay term to `hebbian.py` so that entity pair weights decrease when pairs do not co-activate, preventing the graph from becoming a monotonic accumulator and ensuring edge weights reflect current interest topology rather than cumulative historical co-occurrence.

---

## Non-Goals

- Not replacing the existing ETA=0.3 Hebbian increment (exp-003 validated, keep it)
- Not implementing full Oja's rule / lateral inhibition (separate experiment)
- Not modifying the co-occurrence baseline weights (only Hebbian-written edges)
- Not adding decay to typed semantic relations (uses, depends_on, part_of, etc.)
- Not decaying interest_area associations — only `related_to` edge weights updated by Hebbian

---

## Constraints

- [owner-intent] LLM calls must use `env -u ANTHROPIC_API_KEY claude -p` (subscription billing)
- [owner-intent] Embeddings must remain local (all-MiniLM-L6-v2, never sent to API)
- [owner-intent] Vault root: `~/memory/vault/`, `graph.db` is a symlink to active snapshot
- Decay must run in the same `vault chunk` call as the Hebbian increment — not a separate sweep
- Decay must be applied only to edges that were NOT fired in the current session's atom batch
- Decay must not drive weights below 0.0
- Must remain <1s incremental — cannot scan the full graph for every chunk call
- SQLite WAL mode must be preserved; no schema changes to `relations` table

---

## Assumptions

- Entity pairs not fired in the current session are detectable: we know which pairs fired (from `update_from_atoms`), so the complement (pairs not fired) is graph_edges - fired_pairs
- The number of `related_to` edges in graph.db is bounded enough to scan per session (<5000 edges expected at current scale)
- DECAY rate will need empirical calibration — biological analogies suggest 0.01–0.05 per session is plausible but this needs validation
- The Hebbian increment (ETA=0.3) and decay should compound cleanly: `new = min(old + ETA, MAX_WEIGHT)` on fire, `new = max(old * (1 - DECAY), 0.0)` on silence
- Edges created by the co-occurrence baseline (not Hebbian) should be excluded from decay to protect the structural backbone of the graph
- A single DECAY constant is sufficient for a first experiment; per-entity-type decay is out of scope

---

## Open Questions

- What DECAY rate preserves signal for genuinely stable associations while allowing stale ones to fade? Biological LTD suggests 1–5% per "session" — but what constitutes a session in this context changes the effective decay half-life significantly.
- Should decay apply to ALL `related_to` edges or only those with `description='hebbian'`? Applying to all risks destroying the co-occurrence baseline; applying only to hebbian edges requires distinguishing them at read time.
- What is the expected half-life of a strong association? If autoresearch ↔ Cortex co-activates 65 sessions, how many non-firing sessions should it take to decay to half weight? This defines DECAY.
- Does decay interact with the MAX_WEIGHT ceiling in a pathological way? Pairs that frequently hit MAX_WEIGHT may "bounce" between MAX_WEIGHT and a decayed value in a visible oscillation pattern.
- How do we measure whether decay improves graph quality? We need a before/after metric — community stability, recall precision, or a held-out probe set.
- Is scanning all `related_to` edges per session feasible? At 2548 edges (current production graph), a full scan per chunk call costs ~2ms — acceptable. But what's the growth trajectory?

## Next Research Steps

1. Survey biological Hebbian decay models — BCM theory, STDP, homeostatic plasticity — to ground the DECAY constant in neuroscience rather than guessing.
2. Investigate existing graph weight decay implementations in knowledge graph systems (Zep, Graphiti, MemGPT) — do any implement temporal decay on edges?
3. Determine best practice for distinguishing Hebbian-written edges from structural co-occurrence edges in the `relations` table — schema options (flag column, description field, relation_type subtype).
4. Find empirical half-life guidance: how quickly do human interests actually shift? What timescale would make DECAY meaningful for a personal memory system used daily?
5. Identify evaluation metrics for decay quality — what does "the graph encodes current topology" look like quantitatively vs qualitatively?
