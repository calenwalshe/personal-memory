# Clarify Brief: dev-memory-feel

**Slug:** dev-memory-feel
**Timestamp:** 20260414T140000Z
**Status:** draft
**Complexity:** standard

---

## Idea

What does "feeling known" mean specifically for developer/technical work in Claude Code — and what is the minimum viable, testable expression of that phenomenology?

*(Second iteration following memory-phenomenology concept research. The prior dossier established that no AI memory system has defined a phenomenological target, and that the central question for this system — what "being known" means for a developer — was not answered. This brief frames that question specifically.)*

---

## Goal

Produce a concrete, testable definition of what "feeling known by Claude Code" looks like for a developer — expressed as 3-5 specific workflow moments where memory would be felt, not just retrieved — so the memory system can be designed and evaluated against these moments rather than against extraction metrics.

---

## Non-Goals

- Not a UI/UX specification for any new command or interface
- Not a comparison of AI memory architectures (covered in memory-phenomenology)
- Not a redesign of the extraction pipeline or window classifier
- Not about general/personal memory (lifestyle, emotional support) — only developer/technical work memory
- Not about what the memory system *stores* — only about when and how it would be *felt*

---

## Constraints

- Local embeddings must remain local (privacy invariant — no embeddings sent to API)
- Vault operates independently from Cortex state
- Must work within Claude Code's session model — no persistent process between sessions
- Any "felt" experience must be achievable without model fine-tuning or changes to how Claude works fundamentally
- Current vault stores semantic facts (atomic), not episodic context — any proposed "moments" must be achievable with what can be extracted today or with near-term extensions to fact_store.py
- The current entry point is entirely opt-in (`/recall`) — felt experience cannot currently happen without user initiation

---

## Assumptions

- Developer context is categorically different from general assistant context: the relevant memories are code preferences, architectural decisions, debugging patterns, tool choices — not lifestyle or emotional history
- The felt experience of being known as a developer = the assistant demonstrating understanding of *how you work* and *what you're building*, not who you are personally
- A minimal viable "feeling known" moment can be constructed from even a sparse vault (5-10 sessions), without requiring months of history
- The most impactful developer memory moments are not generic preferences ("prefers Python") but project-specific context ("has been refactoring the auth middleware for 3 sessions and is blocked on the session token issue")
- Proactive surfacing (assistant mentions memory without being asked) is the mechanism that produces felt continuity — passive retrieval via `/recall` produces information access, not felt recognition

---

## Open Questions

- q1 (codebase): What types of facts is the personal-memory system actually extracting from Claude Code sessions today — are they developer-specific or generic?
- q2 (factual): What are the 3-5 concrete moments in a developer's workflow where memory would feel most impactful — and what does "impactful" mean in each case?
- q3 (factual): What is the taxonomy of developer-specific memory categories and their relative importance to felt experience? (code style, architectural decisions, debugging patterns, tool preferences, recurring errors, project context)
- q4 (landscape): How do developer-focused AI tools (GitHub Copilot, Cursor, Windsurf, Zed) handle persistent context about the developer's working style?
- q5 (comparison): What is the right format for session-start memory injection — CLAUDE.md-style static file, runtime prompt prefix, or explicit /recall surface?
- q6 (factual): What is the minimum viable memory demonstration — what would "feeling known" look like with only 5-10 sessions of history, before the vault is deep?

---

## Next Research Steps

1. (q1/codebase) Audit the personal-memory vault: sample extracted facts from `facts.db`, classify by type (generic vs developer-specific, semantic vs episodic), and determine what the current system actually produces.
2. (q4/landscape) Survey how developer-focused AI tools (Copilot, Cursor, Windsurf, Zed) handle persistent developer context — what they actually remember, how they surface it, what the felt developer experience is.
3. (q2+q6/factual) Identify 3-5 concrete "minimum viable memory moments" — specific workflow situations where memory would be felt, expressible as: "The assistant says X when Y happens, referencing Z from a prior session."
4. (q3/factual) Define a taxonomy of developer-specific memory categories ordered by their impact on felt experience — project context > architectural decisions > debugging patterns > code style preferences > tool preferences.
5. (q5/comparison) Compare session-start injection approaches for developer context: CLAUDE.md injection (persistent, static), runtime prefix (dynamic, session-aware), explicit surface (/recall) — and what each feels like to the developer.
