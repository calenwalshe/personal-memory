# Clarify Brief: knowledge-consolidation-engine

**Slug:** knowledge-consolidation-engine
**Timestamp:** 20260419T030000Z
**Status:** approved
**Complexity:** complex

---

## Idea

Redesign the SCAPE memory vault from a chat-only capture system into a universal knowledge consolidation engine. Three changes: (1) L0 becomes a universal intake adapter accepting any source type — notes, docs, PDFs, Cortex briefs, fact dumps, code logs — not just Claude sessions. (2) L1 atoms become evidence units with source provenance and candidate labels (obs:plan, obs:decision), not final truth assertions. (3) A new L3 belief runtime engine sits above L2, tracking what the system currently believes, what used to be true, what's planned, what's contested — with Kripke-inspired worlds, temporal truth tracking, inference rules, and domain-adaptable modules (PersonalMemory, Cortex, Research) that can be enabled per project.

---

## Goal

The vault transitions from "what happened in chat sessions" to "what do I believe, why, and how confident am I" — with any source type feeding evidence into a modular belief engine that produces stable knowledge, detects contradictions, tracks plans, and derives lessons.

---

## Non-Goals

- Full Datalog/Soufflé engine (pure Python rules sufficient for v1)
- OWL type system or description logic (L2 entity types are enough)
- Event Calculus temporal reasoning (world=past label is sufficient)
- More than 4 starter inference rules (scale rules as use cases emerge)
- Multiple L2 organizers (entity graph stays the only organizer; formalize protocol only)
- L2 run tracking / organizer output tables (YAGNI with one organizer)
- Separate evidence_units table (extend atoms for v1; rename later if needed)
- Full label provenance table (JSON observed_labels for v1; label table for v2)

---

## Constraints

- All SQLite databases use WAL mode, async writes, never block the CLI
- All `claude -p` calls must use `env -u ANTHROPIC_API_KEY` for subscription billing
- Existing atoms.db schema extensions must be backward-compatible (nullable columns only)
- events.db remains untouched — it is the live telemetry stream
- graph.db remains untouched — L2 entity graph, communities, Hebbian decay all stay
- No external dependencies beyond what's already installed (sentence-transformers, faiss, sqlite3)
- Existing hooks (PostToolUse, SessionEnd, PreCompact) continue to work unchanged

---

## Assumptions

- 415 atoms and 1258 facts are a sufficient base for testing L3 extraction and inference
- Haiku can reliably extract typed logical forms (claim, decision, plan, preference, warning, rule) from evidence units
- Pure Python inference rules (conflict detection, supersession, stable promotion, lesson extraction) cover 80% of L3 value without a formal logic engine
- The 8 Kripke worlds (current, past, planned, possible, contested, rejected, user_belief, system_belief) are a sufficient taxonomy for personal memory and Cortex use cases

---

## Open Questions

- How should the existing 1258 facts in facts.db be migrated? (semantic→logical_forms, episodic→evidence-linked observations, procedural→rule_candidate forms, contradictions→derived_objects)
- What is the right extraction trigger for L3? Inline after L1 chunker, or on-demand via `vault l3 extract`?
- Should L3 extraction run on ALL existing atoms (415 backfill) or only new atoms going forward?
