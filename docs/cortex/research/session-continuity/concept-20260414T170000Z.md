## Owner Summary

- **What we found:** The hook infrastructure already exists — PreCompact and SessionEnd both fire, both receive `transcript_path`, and both can read the live conversation. No new platform capability is needed. The gap is purely in what the hooks write.
- **What it changes:** The implementation is a writer, not a new reader. Two hook scripts write to `.cortex/session-memory.md`; the existing SessionStart hook reads and injects it. No FAISS, no LLM at session start, no vault lookup — just file I/O.
- **What's still open:** Whether to call an LLM (Haiku) in the PreCompact hook to summarize "what we're doing right now" — or to write raw recent turns and let session-start injection handle it.

---

## Summary

The personal-memory system already has PreCompact and SessionEnd hooks configured and firing. Both receive `transcript_path` — a pointer to the full session JSONL — giving them direct access to conversation content without FAISS or vault lookups. The two continuity artifacts (snapshot and summary) map cleanly to two existing hook events. The injection surface is the existing SessionStart hook's `additionalContext` mechanism, which already fires at startup, resume, post-clear, and post-compact. The missing piece is a `.cortex/session-memory.md` file that PreCompact writes and SessionEnd updates, which the SessionStart hook then reads and surfaces. Window classifier output is explicitly not sufficient for session summaries — a dedicated Haiku summarization pass on the transcript is needed. The `Stop` hook, which fires on every agent turn, must NOT be used for summary writing.

## Findings by Question

### q1 (codebase): What Claude Code hooks exist?

**Finding:** All required hooks exist and are already configured.

| Hook | When it fires | Payload includes | Currently does |
|---|---|---|---|
| PreCompact | Before context is wiped | session_id, transcript_path, cwd | Writes metadata snapshot to `.cortex/compaction/` and vault breadcrumb to `vault/raw/sessions/` |
| SessionEnd | Once at session termination | session_id, transcript_path, cwd, matcher | Writes `current-state.md` from state.json (via cortex-session-end.sh on Stop hook — WRONG hook) |
| Stop | After EVERY agent turn | session_id, transcript_path | Currently used for cortex continuity writes — fires too frequently |
| SessionStart | On startup, resume, /clear, post-compact | cwd, project | Injects additionalContext (current-state.md, facts, system-map) |

**Critical finding:** `transcript_path` is present in both PreCompact and SessionEnd payloads. Both hooks can read the full live conversation directly — no FAISS, no LLM lookup needed.

**Bug/misconfig finding:** `cortex-session-end.sh` is currently registered on the `Stop` hook, not the `SessionEnd` hook. It therefore fires after every agent response turn, not once at termination. This is likely unintentional and means the session continuity writes are happening on every turn.

**SessionEnd matcher values:** `"clear"`, `"resume"`, `"logout"`, `"prompt_input_exit"`, `"bypass_permissions_disabled"`, `"other"` — the hook can distinguish /clear from normal exit.

Source: code.claude.com/docs/en/hooks, github.com/anthropics/claude-code/issues/34954

### q2 (codebase): What does session-start injection look like today?

**Finding:** The SessionStart hook already injects via `additionalContext` with a 9,500-character budget. It currently reads and injects:
- `docs/cortex/handoffs/current-state.md` (Cortex pipeline state)
- `.cortex/facts.jsonl` (slug-matched prior work facts, semantic retrieval)
- `.cortex/system-map.md` staleness anchor
- Structural graph count and date
- Operational ledger summary
- Coherence warnings from health check

**Vault injection is DISABLED.** The comment in `cortex-session-start.sh` reads: `# DISABLED: vault recall injection — was spawning concurrent FAISS+Claude subprocesses`. This is exactly the problem a pre-written `.cortex/session-memory.md` file solves — read a file, don't spawn a process.

**MEMORY.md at project root is NOT auto-read.** Claude Code does not scan for or inject arbitrary project files. The injection surface is the SessionStart hook's `additionalContext` output. Any session memory must be read by the SessionStart hook and added to that output.

**Implication:** The implementation path is:
1. PreCompact writes `.cortex/session-memory.md` (snapshot content)
2. SessionEnd updates `.cortex/session-memory.md` (summary content)
3. SessionStart reads `.cortex/session-memory.md` and adds to additionalContext

Source: `~/.claude/hooks/cortex-session-start.sh` (direct codebase read)

### q3 (codebase): Is window classifier output sufficient for session summaries?

**Finding: No. A dedicated Haiku summarization pass is required.**

The window classifier extracts atomic facts within 5-turn sliding windows. What it captures:
- Learning moments (question → explanation → confirmation patterns)
- Atomic facts: "FAISS IndexFlatIP requires L2-normalized vectors"
- Timestamped to session date (not intra-session turn)

What is **missing** for a session summary:
- Temporal narrative: "you explored approach A (turns 10-45), then switched to B"
- Session-level intent/goal: what motivated the session
- Work distribution: how much time on which topic
- Causal links between discoveries and decisions
- Sequential flow of what was attempted

The episodic extractor captures events/decisions but also lacks temporal flow and session intent.

**Key insight:** The continuity snapshot doesn't need the window classifier at all. The PreCompact hook has `transcript_path` — it can read the last 20-30 turns of the conversation directly and call Haiku to summarize "what we're doing right now." No vault lookup, no FAISS, just a Haiku call on recent raw turns.

The session summary (SessionEnd) similarly reads the full transcript and calls Haiku to produce a 1-2 paragraph summary. The window classifier continues running independently for long-term vault storage — it doesn't need to change.

Source: `scripts/window_classifier.py`, `scripts/fact_store.py` (direct codebase read)

### q4 (factual): What is the minimum content schema for each artifact?

**Continuity snapshot (PreCompact):**

```
## Session Snapshot — {date} {time}
What we're doing: {1 sentence — current task}
Where we are: {1-2 sentences — current state, what's working/not working}
Next step: {1 sentence — what comes next}
Blocking context: {1 sentence — key constraint or hypothesis, if any}
```

4-5 lines. Immediately actionable. The developer reads this and can jump back in without asking "where were we?"

**Session summary (SessionEnd):**

```
## Session Summary — {date}
Project area: {component/feature/bug being worked on}
What was accomplished: {2-3 sentences — what changed or was decided}
Key decisions: {bullet list, 1-3 items}
Open thread: {1 sentence — what's left, what the next session should pick up}
Status: {resolved | paused | blocked}
```

8-12 lines. Durable for days. The developer reads this at the start of the next session and knows exactly where they left off.

**What both require:** No subjective summary language ("interesting", "important"). Concrete specifics only. Session summary must include an "open thread" that is specific enough to act on.

Source: session-handoff framework patterns, github.com/duke-of-beans/CONTINUITY

### q5 (comparison): Where should session-memory live?

**Options evaluated:**

| Location | Auto-read? | Developer-visible? | System-owned? | Verdict |
|---|---|---|---|---|
| Project CLAUDE.md | Yes (Claude loads it) | Yes | No (developer-authored) | ✗ — must not mix system writes into dev file |
| Project root MEMORY.md | No | Yes | Could be | ✗ — needs hook modification, no advantage over .cortex/ |
| `.cortex/session-memory.md` | No (hook reads it) | Yes (if developer looks) | Yes | ✓ — clean separation, hook reads it |
| `~/memory/vault/raw/` | No | Hidden | Yes | ✗ — too buried, defeats consent goal |
| Global `~/.claude/CLAUDE.md` | Yes | Yes | Risky | ✗ — cross-project noise |

**Recommendation: `.cortex/session-memory.md`**

- System-owned (developer's CLAUDE.md is untouched)
- Visible when developer browses `.cortex/` — inspectable, editable, deletable
- Read by the SessionStart hook via additionalContext — same mechanism as current-state.md
- Not auto-read by Claude Code (no risk of interference with other projects)
- Scoped to the project (correct per-project isolation)

The SessionStart hook already reads files in `.cortex/` and injects them. Adding `session-memory.md` to that injection is a 5-line change.

### q6 (factual): Should behavior differ on /clear vs compaction vs normal exit?

**Finding: Yes — three distinct behaviors, mapped to intent.**

| Event | Intent | Correct behavior |
|---|---|---|
| PreCompact (auto) | Continue working — context is just full | Write continuity snapshot only. Developer wants to keep going. |
| PreCompact (manual) | Explicit compaction — still continuing | Same as auto. |
| SessionEnd (matcher != "clear") | Normal exit — session complete | Write or update session summary. Keep any existing snapshot. |
| SessionEnd (matcher == "clear") | Deliberate reset — developer chose to start fresh | Write session summary (for future reference), mark as "cleared session". Do NOT write continuity snapshot. |

**The /clear distinction matters:** If the developer runs /clear, they are signaling "I want a clean slate." Writing a continuity snapshot that says "resume from here" would contradict that intent. The session summary should still be written (the work happened), but the "next step" field should be suppressed or marked with "session cleared — no resume context."

**Implementation:** SessionEnd hook checks `matcher` value. If `"clear"`, write summary with `status: cleared` and omit the "open thread" / "next step" fields.

## Adjacent Findings (2 — passed filter pipeline)

### 1. cortex-session-end.sh is on the wrong hook — Stop fires every turn, not at session end

**Finding:** The current `cortex-session-end.sh` is registered on the `Stop` hook, which fires after every agent response turn. It is NOT on the `SessionEnd` hook. This means the current-state.md is being rewritten on every single agent turn — not once at session termination as intended. **Why this matters for session-continuity:** The session summary must be registered on `SessionEnd`, not `Stop`, or it will fire 50+ times per session. The existing misregistration should be fixed as part of this work, not worked around. Source: `~/.claude/hooks/cortex-session-end.sh` (direct codebase read), code.claude.com/docs/en/hooks

### 2. transcript_path in PreCompact payload eliminates the need for mid-session vault lookups

**Finding:** Both PreCompact and SessionEnd receive `transcript_path` — a pointer to the live session JSONL file. The hook can read the last 20-30 turns directly and call Haiku for a targeted summarization, with no FAISS index, no vector search, no vault lookup. The previously-disabled vault injection was disabled specifically because it spawned "concurrent FAISS+Claude subprocesses." **Why this matters:** The entire session-memory write architecture can be implemented without touching the vault pipeline — just `jq .content transcript_path | tail -N | haiku "what are we doing?"`. This is a much smaller implementation than the clarify brief assumed. Source: `~/.claude/hooks/cortex-session-start.sh`, code.claude.com/docs/en/hooks

## Synthesis: The Implementation

All six questions resolve to a single, small implementation:

**Two hook scripts + one read in SessionStart:**

1. **`precompact-session-snapshot.sh`** (PreCompact):
   - Reads last 30 turns from `transcript_path`
   - Calls Haiku: "In 4-5 lines: what task is being done, what's the current state, what's the next step"
   - Writes output to `.cortex/session-memory.md` under `## Session Snapshot`

2. **`sessionend-session-summary.sh`** (SessionEnd):
   - Reads full transcript from `transcript_path`
   - Calls Haiku: "In 8-12 lines: what was worked on, key decisions, open thread, status"
   - If `matcher == "clear"`: suppresses open thread, sets `status: cleared`
   - Writes/replaces `.cortex/session-memory.md` under `## Session Summary`
   - Also fixes: move `cortex-session-end.sh` from Stop → SessionEnd

3. **SessionStart hook modification** (5 lines):
   - Add: read `.cortex/session-memory.md` if it exists, inject into additionalContext
   - Budget: 500-800 characters (short content from the schema above)

**What this is NOT:**
- Not a redesign of the vault or window classifier
- Not a new storage format
- Not a FAISS or embedding change
- Total scope: ~150 lines of bash + one Haiku call per compaction/exit event
