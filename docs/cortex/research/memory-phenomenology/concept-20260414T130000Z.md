## Summary

Every AI memory system in the field (Letta, Mem0, Zep, Rewind) describes itself in architectural terms — token efficiency, temporal knowledge graphs, hybrid storage — with no documented phenomenological target. A Gemini adversarial challenge confirmed the gap: these systems answer "how does the AI store memory?" but not "what does it feel like to interact with a system that remembers you?" Human memory research provides the sharper frame: Endel Tulving's 1972 episodic/semantic distinction — *remembering* (a feeling of conscious reliving) vs. *knowing* (fact retrieval without context) — maps directly onto the difference between an assistant that makes you feel understood and one that merely retrieves data about you. The personal-memory system's codebase analysis reveals the current F1 metric measures binary window classification accuracy, which is orthogonal to whether any extracted fact ever re-shapes a future session. The most damaging failure mode is not forgetting but confident wrong memory ("context hallucination") — and raising recall naively would flood the vault with low-signal facts that erode trust faster than silence would. The three design principles that survive the research: precision over recall, temporal metadata is load-bearing, and proactive surfacing (the assistant mentions memory unprompted) is the mechanism through which "being known" is experienced.

## Findings

- **No phenomenological design goal exists anywhere in the field.** Letta, Mem0, and Zep describe architecture and integration benefits (token reduction, developer control, temporal accuracy); none has a documented statement of what the target user experience should feel like. This is not a gap in the literature — it is a gap in industry practice. [Perplexity + Gemini challenge]

- **Confident wrong memory is the primary trust-damaging failure mode, not forgetting.** "Context hallucination" (filling gaps with plausible fabrications) and "context staleness" (presenting outdated facts as current) are more damaging than a simple gap. When an AI says "I know you prefer X" incorrectly, it manufactures a false identity for the user — a qualitatively different harm from silence. [Tavily, chrislema.com]

- **Tulving's episodic/semantic distinction is the right frame.** Episodic memory (timestamped, contextual, felt as reliving) builds sense of self; semantic memory (decontextualized facts) does not. The current system extracts primarily semantic facts. A system that only stores "user prefers async/await" provides *knowing*, not *remembering* — the difference Tulving identified between fact retrieval and the phenomenal feeling of continuity. [simplypsychology.org, tianpan.co]

- **The temporal dimension of memory is load-bearing.** "The user said they were evaluating the product" means something different depending on when it was said. Without timestamps and temporal reasoning, recalled facts can mislead. Temporal questions are the hardest in LoCoMo benchmarks (Mem0: 55.5%, Zep: 79.8%), and most memory systems struggle here because they store facts without preserving chronological structure. [tianpan.co, byterover.dev]

- **Hybrid memory grain wins: atomic facts + narrative context together.** Systems treating all memory uniformly fail — semantic triples alone produce "data-rich but insight-poor" outputs, while raw conversation history becomes noisy at scale. Linking atomic facts to their originating episodic summaries enables both precise retrieval and coherent reasoning. PersonaMem-v2 style routing (raw, fact, episodic layers) reduces tokens 16× while improving personalization. [Perplexity, decodingai.com]

- **Current F1 is orthogonal to phenomenological success.** F1 measures binary classification accuracy (learning moment vs. task execution) on 50 probe examples, with current performance of 0.25 (recall=0.167). It does not measure: fact extraction quality, phenomenological salience, whether facts re-surface in future sessions, or confidence calibration. A system achieving F1=0.90 could still produce zero phenomenological value if extracted facts never shape future session context. [Codebase analysis, scripts/window_classifier.py, scripts/GOAL.md]

- **Raising recall naively is the wrong direction.** F1=0.25 with recall=0.167 means 83% of learning moments are missed — but loosening the confidence threshold to catch more windows floods the vault with low-signal facts. Since confident wrong memory is the worst failure mode, higher vault noise degrades the user experience faster than a sparse vault does. Precision should be the primary optimization target.

- **Forgetting is a feature, not a failure.** Spaced replay frameworks (FOREVER, Anki-style systems) model memory as having natural decay curves. An AI memory system without a forgetting policy accumulates noise; stale facts are cited as current. Forgetting low-signal memories after 60-90 days of inactivity prevents vault rot. The user should be able to see which memories are decaying — transparency in decay builds trust. [Perplexity on forgetting curves]

- **The "being known" experience requires proactive surfacing, not passive retrieval.** The current `/recall` architecture puts all responsibility on the user to ask. Phenomenological continuity — the felt sense that the assistant knows your context — requires the system to mention or apply memory without being prompted. This is a session-start injection problem, not a query-time retrieval problem.

- **Memory bleeding across contexts is a trust and privacy failure.** Users need context-scoped memory, not global recall. A health disclosure surfacing in a work-email context is not just an unwanted preference — it is a trust violation. The current vault has no per-context scoping. [techpolicy.press]

## Trade-offs

### Option A: Precision-first extraction (raise confidence threshold, accept lower recall)
**Pros:** Fewer false positives; surfaced memories are trustworthy; avoids vault noise; aligns with "confident wrong memory is the worst failure mode" finding.
**Cons:** Vault grows slowly; many learning moments missed; user may not notice memory working at all.
**Verdict:** recommended — precision is the correct primary target given the trust dynamics.

### Option B: Recall-first extraction (lower threshold, flood vault)
**Pros:** Vault grows faster; more learning moments captured.
**Cons:** Confident wrong memory erodes trust more than a gap; vault noise degrades recall quality; contradicts primary phenomenological failure mode finding.
**Verdict:** rejected.

### Option C: Episodic-first memory (store contextual summaries, not just atomic facts)
**Pros:** Builds the felt sense of continuity (Tulving's *remembering*, not just *knowing*); temporal dimension preserved; episodic + semantic hybrid is the research-validated winning pattern.
**Cons:** Requires architecture changes to current fact_store.py; more storage; more complex retrieval.
**Verdict:** deferred — informs spec direction, not current implementation sprint.

### Option D: Proactive session-start injection (surface 2-3 facts at session start)
**Pros:** Directly delivers the phenomenological goal (user feels known without asking); no change to extraction pipeline required — works on existing vault.
**Cons:** Requires integration with Claude Code session hooks; risk of surfacing wrong/stale facts confidently (uncanny valley); needs confidence gating.
**Verdict:** deferred — high-leverage but requires precision guarantee first (Option A prerequisite).

## Recommendations

- **Adopt a testable phenomenological design goal:** "The user should occasionally feel their assistant understood something about them without being asked." Concrete test: present the assistant with a new session and check whether it spontaneously applies knowledge from prior sessions without a `/recall` prompt. This replaces F1 as the north-star metric.

- **Define a downstream success metric — fact re-use rate:** Track the % of extracted facts that are referenced in future sessions within 30 days. A fact extracted but never surfaced has zero phenomenological value regardless of F1. This metric directly measures integration, not extraction.

- **Raise the confidence threshold on window_classifier.py:** Prioritize precision. Accept lower recall in exchange for higher trust in surfaced memories. Current threshold sweep (0.3–0.7) should be evaluated against precision, not F1.

- **Add temporal metadata to all extracted facts:** Every fact in facts.db should carry a created_at timestamp and a last_seen_at field. Temporal queries ("what did I know about X in March?") require this. Without it, recall misleads.

- **Design explicit memory decay:** Facts inactive for 60-90 days should be marked low-priority or archived, not deleted. Forgetting is a feature. The vault should have a visible "freshness" signal per fact, surfaced in `/recall` output.

- **Investigate session-start injection as the highest-leverage phenomenological lever:** The current system is entirely opt-in (user must type `/recall`). Proactive surfacing — injecting 2-3 high-confidence, recent facts into the session start context — is likely the mechanism that produces the felt sense of "being known." This should be the next clarify slug after this research is synthesized.

## Adjacent Findings

- **The uncanny valley of AI memory:** A memory system that recalls almost correctly but slightly wrong is more trust-damaging than one that simply doesn't remember. When a system confidently says "I know you prefer X" but the user moved on from X six months ago, it creates the specific discomfort of being misrepresented by something that presumes intimacy. For this system's design choices, this means precision must gate confidence — it is better to surface nothing than to surface a stale or wrong memory with confidence. Source: chrislema.com/ai-context-failures-nine-ways-your-ai-agent-breaks/

- **Memory as a governance surface:** Users need granular control over what an AI remembers in which contexts — not just a global on/off toggle. The inability to scope memories (e.g., "only recall health context for health conversations") is simultaneously a safety failure, a trust failure, and a privacy failure. For the personal-memory vault, this implies per-project or per-context scoping of facts is a trust requirement, not a nice-to-have. Source: techpolicy.press/we-are-not-talking-about-ai-memory-enough

- **Benchmark fragmentation makes external comparison meaningless:** LoCoMo (the industry standard) reports human performance at 87.9% Token-Overlap F1, while memory systems claim 60-92% using incompatible criteria. The personal-memory system's F1=0.25 measures a different task (window classification) than any external benchmark. There is currently no external grounding for the F1 target — the correct benchmark is not yet chosen, and choosing it is itself a design decision with phenomenological implications. Source: reddit.com/r/MachineLearning + emergentmind.com/topics/locomo
