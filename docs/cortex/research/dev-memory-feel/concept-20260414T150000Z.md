## Executive Summary

The personal-memory vault already contains 568 developer-specific facts including 120 episodic records — the gap is not extraction but surfacing. "Feeling known" for a developer does not require months of history or proactive mid-session interruptions. It requires one precisely-chosen fact surfaced at session start, the highest-ROI moment in the developer workflow because it is already the peak interruption-recovery point (23-45 min recovery time empirically). The MVP is achievable with existing vault data. The path to implementation runs through CLAUDE.md injection — the existing session-start context mechanism — not new infrastructure. Critical constraint: trust and consent must be solved before phenomenology; 81% of developers report security/privacy concerns about AI tools, and "being known" flips to surveillance anxiety if the developer does not control what is stored.

## Findings by Question

### q1 (codebase): What is the vault actually extracting today?

**Finding:** The vault is already developer-specific. 568 total facts: 341 learning moments (60%), 120 episodic (21%), 70 procedural (12%), 37 technical (7%). All categories are developer-context facts, not generic personal preferences. Window classifier F1=0.25 (recall=0.167) means learning moments are under-extracted — 1 captured per ~6 that occur. The extraction gap is significant; the categorization gap is not.

**Implication for "feeling known":** The limiting factor is surfacing, not storage. The system has material to work with already. The MVP demonstration does not require improving extraction — it requires wiring existing vault output to session-start injection.

---

### q2 (factual): What are the 3-5 highest-impact developer workflow moments?

**Finding:** Three workflow moments dominate by recurrence and emotional salience:

1. **Session start after a break** — picking up a codebase after hours or days away. This is the highest-frequency high-friction moment in developer work. Memory that says "last time you were stuck on X, having just refactored Y" directly addresses the 23-45 minute recovery window. This is where vault facts have the highest marginal utility.

2. **Multi-session debugging** — returning to a bug that was not resolved in the previous session. Memory of what was tried, what was ruled out, and what the leading hypothesis was converts a cold restart into a warm continuation. Felt as "the assistant remembers what we already tried" — direct recognition signal.

3. **Code review of one's own patterns** — when the assistant references a preference or constraint the developer established in a prior session without being prompted ("you mentioned preferring immutable data structures in the auth module"). This is the clearest phenomenological "being known" moment, but it requires a fact that actually recurs in the work.

Sources: super-productivity.com/blog/context-switching-costs-for-developers, basicops.com/blog/the-hidden-cost-of-context-switching

### q3 (factual): What is the taxonomy of developer memory categories?

**Finding:** Two-tier taxonomy, ordered by impact on felt experience:

**Tier 1 — Project context (highest impact, recurs across sessions, affects output immediately):**
- Active work state: what was being built, what was blocked, where progress was left
- Architecture decisions: constraints and rationale for how the system is structured
- Infrastructure conventions: ports, deployment targets, environment configurations
- Tech stack specifics: frameworks, library versions, build tooling

**Tier 2 — Personal/working style (moderate impact, stable, affects tone and approach):**
- Code quality standards: "readability over brevity," optimization philosophy
- Decision-making preferences: how to handle ambiguity, when to ask vs. proceed
- Communication style: response length, detail level, preferred framing

**Ordering conclusion:** Project context > architectural decisions > debugging patterns > code style preferences > tool preferences. This ordering is consistent with the memory-phenomenology finding that episodic/contextual facts produce felt recognition while decontextualized semantic facts produce data retrieval.

The TDS article confirms: "The most impactful memories are those that recur across sessions, affect output quality immediately, and travel with the codebase."

Sources: towardsdatascience.com/why-every-ai-coding-assistant-needs-a-memory-layer, tribe.ai/applied-ai/beyond-the-bubble-how-context-aware-memory-systems-are-changing-the-game-in-2025

### q4 (landscape): How do Copilot, Cursor, Windsurf, Zed handle persistent developer context?

**Finding:** No competitor has cross-session episodic memory for developer workflow. All approaches are either:
- **Static config files** (CLAUDE.md-style): developer manually writes context; persistent but not auto-updated
- **Within-session context windows**: full conversation history within one session; lost on close

The gap this system could fill — the one no competitor occupies — is **automatically extracted, session-persistent episodic memory** that does not require the developer to author it manually. This is a genuine product gap, not a crowded space.

**Implication:** The competitive reference point is CLAUDE.md. The question is not "which approach is better?" but "what does CLAUDE.md not do that the vault could add?" Answer: CLAUDE.md captures static developer-authored preferences; the vault captures dynamic project-state that changes session to session.

---

### q5 (comparison): CLAUDE.md vs dynamic injection vs /recall — what feels like being known?

**Primary finding:** CLAUDE.md is the existing injection mechanism. It already runs at session start. Vault facts can be injected into CLAUDE.md (or a CLAUDE.md-adjacent file) without new infrastructure. The distinction:
- Static CLAUDE.md: developer writes once, persists indefinitely
- Dynamic vault injection: system writes on session end, reads on session start, facts expire

**Gemini challenge finding:** The real gap is not choosing between formats — it is the **missing hybrid**: vault facts that can be promoted to CLAUDE.md via explicit developer approval, with confidence signals ("high confidence — appeared in 4 sessions") and staleness markers ("last seen 30 days ago"). This closes the loop between automatic extraction and developer-controlled permanence.

**Key insight from challenge:** The injection format is not the primary variable. The primary variable is **what gets surfaced and when**. A 3-fact CLAUDE.md injection at session start is more valuable than a 50-fact lookup via /recall, because the developer does not have to initiate it and the surface is already familiar.

---

### q6 (factual): What is the minimum viable "feeling known" demonstration?

**Finding:** MVP requires 3 conditions, not a deep vault:

1. **At least 3 sessions of context** — enough to establish pattern (not random)
2. **One well-chosen episodic fact at session start** — "last session you were blocked on X" produces recognition; "you prefer Python" produces data retrieval
3. **Correct precision** — one confident, accurate fact > five hedged facts. Confident wrong memory is the worst failure mode (from memory-phenomenology)

With 120 episodic facts already in the vault from existing sessions, the demo is achievable with current data. The implementation path:
1. Surface the top-1 most recent episodic fact at session start via CLAUDE.md injection
2. Express it with temporal context ("last session, you were debugging the session token issue in auth middleware")
3. Measure: does the developer respond as if they recognize it?

This is the minimum viable experiment. It requires no new extraction, no new storage, no new infrastructure — only a read from facts.db and a write to the CLAUDE.md context file.

## Adjacent Findings (3 — passed filter pipeline)

### 1. Session start is the peak context-switch recovery moment — memory ROI is highest there

**Finding:** Empirically, developers require 23-45 minutes to recover flow state after an interruption, with complex coding tasks at the upper bound. Session start after a break is already the interruption-recovery moment in Claude Code. Memory injection at session start is not cosmetic — it directly addresses one of the highest-cost, highest-frequency events in the developer workflow. **Why this matters for dev-memory-feel:** It means the MVP moment (q6) is not an invented use case — it is the highest-leverage real one. Any "feeling known" demo should target session start specifically, not mid-session injection. Source: super-productivity.com/blog/context-switching-costs-for-developers, basicops.com/blog/the-hidden-cost-of-context-switching

### 2. Trust/consent gap precedes phenomenology — "being known" can invert to surveillance

**Finding:** 81% of developers report security/privacy concerns about AI tools (Stack Overflow 2025 survey). 46% actively distrust AI accuracy. For a memory system storing developer-specific patterns and decisions, the "feeling known" phenomenological target has a failure mode: if the developer does not understand or control what is being stored, the experience inverts from recognition to surveillance. **Why this matters for dev-memory-feel:** Any spec for the memory UX must address consent design — what gets stored, how to see it, how to delete it — before addressing injection format. CLAUDE.md injection specifically should be readable and editable by the developer, not written invisibly. Source: survey.stackoverflow.co/2025/ai, blackduck.com/blog/ai-coding-assistant-security-risks-benefits-devsecops-2025

### 3. Temporal decay is missing from the taxonomy — confidence without staleness handling replicates the worst failure mode

**Finding:** The q3 taxonomy (project context > architectural decisions > debugging patterns > style preferences) does not differentiate by staleness rate. Active work state decays in days. Architectural decisions decay in months. Code style preferences are nearly permanent. A memory system that injects "you've been working on the auth refactor" two months after the refactor completed is not just unhelpful — it erodes trust faster than no memory at all, because it demonstrates confident wrongness (the worst failure mode identified in memory-phenomenology). **Why this matters for dev-memory-feel:** The injection mechanism must carry staleness signals. Facts with age > threshold should be surfaced with qualification ("as of 3 weeks ago...") or suppressed, not confidently asserted. This is a spec requirement, not a nice-to-have. Source: memory-phenomenology dossier (precision over recall finding)

## Synthesis: The Revised Model

The critical correction from this research pass: the clarify brief's tension between "proactive surfacing is the mechanism" and "current system is opt-in only" is not a contradiction to resolve — it is a product decision to make. Two viable paths exist:

**Path A — Instrument the existing mechanism:** Inject vault facts into CLAUDE.md (or a CLAUDE.md-adjacent file) automatically at session end. The developer does nothing new; the content is already there at session start. This is technically proactive (the assistant references vault context without being asked) while being architecturally opt-in (facts are written to a visible file the developer can read/delete). This collapses the contradiction.

**Path B — Make /recall feel like recognition:** Redesign the /recall UX so that calling it produces episodic framing ("last session you were debugging X, here's what was tried") rather than semantic retrieval ("here are facts tagged auth"). Same data, different phenomenology. This is a lower-lift change but produces a weaker felt experience than Path A.

The research evidence strongly favors Path A — CLAUDE.md injection with staleness signals and developer-visible/editable output — as the mechanism that can achieve the phenomenological goal within the stated constraints.
